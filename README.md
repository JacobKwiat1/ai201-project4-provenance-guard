# Provenance Guard

A backend API for creative sharing platforms to determine whether text content was written by a human or generated by AI. It runs a two-signal detection pipeline, returns a human-readable transparency label alongside a confidence score, supports creator appeals, and maintains a structured audit log of every decision.

---

## Endpoints

| Method | Path | Description |
|---|---|---|
| `POST` | `/submit` | Submit text for AI/human attribution |
| `POST` | `/appeals` | File an appeal against a prior decision |
| `GET` | `/audit/log` | Retrieve the structured audit log |

---

## Detection Signals

The system uses two independent signals that are combined into a single confidence score. They were chosen specifically because they are complementary — each catches what the other misses.

### Signal 1 — LLM Classification (weight: 0.6)

The text is sent to `llama-3.3-70b-versatile` via Groq with a system prompt that instructs it to act as a forensic text analyst. The model assesses stylistic coherence, semantic naturalness, and structural patterns, then returns a structured JSON verdict: `result` ("ai" or "human") and `confidence` (0.0–1.0).

**Why this signal:** A large language model can evaluate meaning and style holistically in a way no rule-based system can. It picks up on things like hedging language, over-explanation, and formulaic transitions that are characteristic of AI output — properties that are hard to quantify but obvious to a reader. It carries the higher weight (0.6) because it's the more reliable of the two signals.

**What it cannot detect:** AI text that is deliberately crafted to mimic a specific human voice, or casual human writing that happens to use structured phrasing. It also reflects training data biases — it may penalize formal academic writing as AI-like because that style overlaps with common AI output patterns.

### Signal 2 — Stylometric Heuristics (weight: 0.4)

Three statistical properties of the text are computed and combined into an AI-likelihood score:

- **Sentence length variance** — AI writing tends to produce uniformly structured sentences; human writing varies more. Low variance scores as AI-like.
- **Type-token ratio (TTR)** — the ratio of unique words to total words. Higher TTR indicates more diverse vocabulary, slightly characteristic of AI systems that avoid repetition.
- **Punctuation density** — AI-generated text often includes structured punctuation (commas, colons) more consistently than casual human prose.

Each metric is mapped to a 0–1 AI-likelihood score and combined with weights (variance 0.5, density 0.3, TTR 0.2). The composite `style_score` is used directly as the signal's AI-likelihood contribution.

**Why this signal:** It is model-independent. It doesn't require an API call or understand meaning — it measures mathematical properties of the text. This makes it a useful cross-check on the LLM signal: if both agree, confidence is high; if they disagree, the system surfaces uncertainty rather than amplifying one signal's error.

**What it cannot detect:** Meaning, tone, or intent. It also degrades significantly on short texts (fewer than ~150 words) because statistical properties don't stabilize at low sample sizes. Two sentences can have high variance simply because they happen to be different lengths, not because of genuine stylistic diversity.

**What I'd change for real deployment:** Replace the hand-tuned metric thresholds with ones calibrated on a labeled dataset of confirmed AI and human texts. The current thresholds (e.g. sentence length variance normalized to /60) were set by reasoning about typical values and testing against a handful of examples — that's fragile. I'd also add a fourth metric: average sentence complexity (clause depth or parse tree height), which more reliably separates AI's tendency toward grammatically complete but shallow sentences from human writing's run-ons and fragments.

## Confidence Scoring

Both signals are converted to a common AI-likelihood scale (0 = human, 1 = AI) and combined via weighted average:

```
ai_likelihood = 0.6 × llm_ai_score + 0.4 × stylometric_score
```

The final attribution and confidence score are determined by thresholds:

| ai_likelihood | attribution | confidence_score |
|---|---|---|
| ≥ 0.75 | `ai` | ai_likelihood |
| ≤ 0.25 | `human` | 1 − ai_likelihood |
| 0.25 – 0.75 | `uncertain` | max(ai_likelihood, 1 − ai_likelihood) |

**Why a weighted average rather than majority vote:** A majority vote between two signals is binary and throws away confidence information — a 0.51 stylometric score and a 0.99 LLM score would vote identically to two 0.51 scores. The weighted average preserves the degree of agreement and lets the higher-weight signal pull the result when signals disagree.

**Why 0.75 as the threshold:** A 75% combined score requires both signals to be broadly aligned. If the LLM says "ai" at 0.9 but stylometrics score only 0.5, the combined score is 0.74 — just below the threshold, producing `uncertain` rather than a confident wrong answer. This is intentional: the system is designed to be conservative, surfacing borderline cases to human review rather than labeling them.

### Example submissions

**High-confidence case — structured AI text:**

> "Machine learning models require large datasets to achieve optimal performance. Data preprocessing is essential to ensure high-quality inputs for training. Feature engineering improves model accuracy by selecting relevant variables. Hyperparameter tuning helps optimize the learning rate and regularization terms. Cross-validation techniques prevent overfitting and improve generalization."

| Signal | Result | Score |
|---|---|---|
| LLM classification | ai | 0.90 |
| Stylometric | ai | 0.87 |
| **Combined** | **ai** | **0.89** |

Both signals agree strongly. The stylometric score is high because all five sentences are nearly identical in length (very low variance) and the vocabulary is dense and non-repetitive. Label returned: `high_confidence_ai` at 89%.

---

**Lower-confidence case — lightly edited AI / borderline text:**

> "I've been thinking a lot about remote work lately. There are genuine tradeoffs — flexibility and no commute on one side, isolation and blurred work-life boundaries on the other. Studies show productivity varies widely by individual and role type."

| Signal | Result | Score |
|---|---|---|
| LLM classification | human | 0.80 |
| Stylometric | ai | 0.74 |
| **Combined** | **uncertain** | **0.59** |

The LLM reads this as human-written (informal opener, personal framing), while stylometrics flags the structured contrast and the final "Studies show..." sentence as AI-like. The signals disagree, so the combined ai_likelihood (0.41) falls in the middle band. Label returned: `uncertain` at 59%.

## Transparency Labels

Every `/submit` response includes a `transparency_label` field for display to readers on the platform. There are exactly three variants:

### `high_confidence_ai`
**Condition:** Combined confidence score ≥ 0.75 and attribution = AI.

**Display text:**
> "Our AI detection system has determined this post to be AI-generated with a high confidence level (N%)."

### `high_confidence_human`
**Condition:** Combined confidence score ≥ 0.75 and attribution = human.

**Display text:**
> "Our AI detection system has determined this post to be human-made with a high confidence level (N%)."

### `uncertain`
**Condition:** Combined confidence score < 0.75 — signals conflict or are individually weak.

**Display text:**
> "Our AI detection system was unable to determine whether this post was AI-generated. Confidence level: N%."

`N` is the confidence score expressed as a rounded integer percentage. The `uncertain` label is intentional for borderline cases: it communicates honest ambiguity to the reader rather than a low-confidence positive or negative verdict. Admins can use the raw confidence score in the audit log to prioritize human review of borderline cases.

## Rate Limiting

Rate limiting is applied per IP address on write endpoints. Read endpoints (`GET /audit/log`) are not rate-limited.

### `POST /submit`

| Window | Limit |
|---|---|
| Per minute | **5 requests** |
| Per hour | **30 requests** |
| Per day | **100 requests** |

**Reasoning:** Each submission triggers a live LLM call (1–3 seconds of real processing time) plus stylometric analysis. The limits are designed around actual writer behavior:

- **5/minute** — A human pasting and submitting text manually can realistically make 2–3 requests per minute. 5 gives comfortable headroom for rapid iteration without friction. A flooding script exhausts this within seconds.
- **30/hour** — Covers a full working session. A writer editing and resubmitting the same piece many times is still well within this ceiling.
- **100/day** — Roughly 3–4 productive sessions. A legitimate user never hits this; an automated scan of thousands of posts would be stopped well before scale.

When a limit is exceeded the API returns `429 Too Many Requests`.

## Known Limitations

### Short texts produce unreliable stylometric scores

The stylometric signal measures statistical properties (sentence length variance, type-token ratio, punctuation density) that only become meaningful with sufficient sample size. On texts under roughly 150 words — a tweet, a product review caption, a short poem — these properties don't stabilize. Two sentences can produce high variance simply because they happen to be different lengths, not because the author varies their rhythm. Three sentences with diverse vocabulary will have a high TTR not because of AI authorship but because short texts inherently repeat less.

This is a structural property of the signal, not a data problem. Adding more training data wouldn't fix it — the math requires more sentences to produce meaningful statistics. In practice, texts under ~100 words will frequently land in the `uncertain` band regardless of origin, not because the system is unsure but because the stylometric component has nothing reliable to contribute.

**What this means for the platform:** Short-form content (captions, taglines, brief comments) should either be excluded from analysis or routed directly to `uncertain` with a flag indicating insufficient length, rather than running through the full pipeline and producing a misleading confidence score.

### AI text written in a casual register evades both signals

The LLM classifier is calibrated on signals like hedging language, formulaic transitions, and over-explanation. When AI output is prompted to write in an informal, personal style — lowercase, sentence fragments, colloquialisms — those markers disappear. The stylometric signal would also score such text as human-like: high sentence length variance (fragments and run-ons), low punctuation density, natural TTR. Neither signal is looking at the right properties to catch this class of AI output, and the combined score would likely fall in the `uncertain` band or incorrectly attribute the content to a human.

## Spec Reflection

**Where the spec helped:** The decision to separate `attribution` from `confidence_score` in the response shape — rather than returning a single label — was specified early and shaped the entire pipeline architecture. It forced the confidence scorer to produce two independent values (a directional judgment and a strength of that judgment) rather than collapsing them into one. This is what makes the `uncertain` band possible: a system that only returns a label can't express "I don't know," but one that separates attribution from confidence can. The spec framing prevented a design decision that would have been hard to undo.

**Where the implementation diverged:** The spec defined a `ContentStatus` model (`"decided" | "under_review"`) as a field that would appear in submission responses and be queryable. In the implementation, `status` is only written to the in-memory `_decisions` store when an appeal is filed — it doesn't appear in the initial `/submit` response, and it isn't returned by `/audit/log` on the analysis entry. The appeal audit entry carries the status instead. This divergence happened because the status field is only meaningful in the context of an appeal: a submission is implicitly `decided` until challenged, and surfacing that in every response adds noise for the common case. If the platform needed to query "show me all content under review," a queryable status field would matter — but the current use case is audit logging, not state management.

## AI Usage

### Instance 1 — Stylometric calibration thresholds and scoring bug

I directed the AI to implement `classify_with_stylometrics()` based on the spec's list of metrics (sentence length variance, type-token ratio, punctuation density). It produced a working function using `/100` as the variance normalization denominator and `0.4` as the TTR lower bound.

When I tested it against four deliberately chosen inputs, the signal returned `"ai"` for every text including clearly human-written casual prose. The issue was twofold: the `/100` denominator meant human texts needed variance above 50 to score as human-like (they typically reached 30–45), and the `0.4` TTR baseline was far below the natural range of short texts (which land at 0.86–0.90 regardless of origin), so the TTR score was pinned near 1.0 for everything.

I recalibrated the thresholds to `/60` for variance and `0.75` baseline for TTR after reasoning about where real texts actually land. I also discovered a separate bug: `compute_confidence()` was treating `style_score` as a confidence-in-prediction and flipping it when `result == "human"`, when it's actually already an AI-likelihood value and should be used directly. That double-inversion was inflating the combined score for human texts. Both fixes were applied manually after testing exposed them.

### Instance 2 — Rate limit values

I directed the AI to apply Flask-Limiter to the `/submit` endpoint. It set the limit to `"60 per minute"` — a round number with no stated reasoning. 60 per minute is one request per second, which is faster than the endpoint's own LLM call latency (1–3 seconds) and indistinguishable from automated flooding.

I overrode this with `"5 per minute; 30 per hour; 100 per day"` after reasoning from the actual usage pattern: an LLM call takes 1–3 seconds, a human manually submitting text can realistically make 2–3 requests per minute, and a writer's session behavior (editing and resubmitting over an hour) is what the hourly and daily windows are designed to accommodate. The numbers are derived from behavior, not chosen arbitrarily.
